# Section 1: What Databases Does NetSuite Offer?
PipeLedger’s scope is deliberately narrow and deep: the General Ledger, the Chart of Accounts, and Project records. These three data objects, combined with the dimensional richness NetSuite attaches to every transaction, contain 90% of the financial intelligence a CFO needs. By staying focused on the GL, we avoid the complexity of sub-system integration while delivering the data that matters most to LLMs.

## 1.1 The General Ledger: Core Financial Record
The NetSuite General Ledger is the central financial database. Every posted transaction creates GL entries that record debits and credits against accounts. What makes the GL powerful for AI consumption is the dimensional richness attached to each journal line.

### Standard GL Dimensions


| Dimension | Purpose | Example Values | AI Relevance |
| --- | --- | --- | --- |
| Account (Chart of Accounts) | Categorizes the nature of the transaction (revenue, expense, asset, liability, equity) | 4100 SaaS Revenue, 6200 Marketing Expense, 1200 Accounts Receivable | Fundamental classification; LLMs need account type + name to understand what a number means |
| Subsidiary | Legal entity identification; required in NetSuite OneWorld for multi-entity companies | Acme Corp US, Acme Corp EU GmbH, Acme Corp UK Ltd | Critical for consolidation; LLMs must know which entity a transaction belongs to and in which currency |
| Department | Cost center / organizational function; typically mirrors budget ownership | Engineering, Sales, Marketing, G&A, Customer Success | Answers “who spent it?” — the organizational dimension most CFOs ask about first |
| Class | Profit center / business line / revenue stream; the most flexible standard segment | SaaS, Professional Services, Hardware, Enterprise, SMB | Answers “what business generated it?” — critical for unit economics and segment reporting |
| Location | Physical or virtual location; used for inventory and geographic reporting | San Francisco HQ, London Office, AWS US-East-1, Remote | Geographic and operational dimension; relevant for statutory reporting and regional analysis |
| Period | Fiscal period assignment; NetSuite auto-assigns based on transaction date and fiscal calendar | Jan 2026, Q1 FY2026, FY2026 | Temporal dimension; LLMs need consistent period labeling to perform trend analysis and comparisons |

### Custom Segments (GL-Impacting)
Beyond the standard four dimensions, NetSuite supports unlimited custom segments that can optionally impact the General Ledger. When a custom segment has GL Impact enabled, it becomes a true financial dimension — every posted transaction carries this tag, and financial statements can be generated by this dimension.


| Common Custom Segment | Industry Use | GL Impact? | AI Value |
| --- | --- | --- | --- |
| Product Line | SaaS, Manufacturing, Retail | Yes | Revenue and COGS analysis by product; margin by product for AI |
| Project / Job | Professional Services, Construction, Biotech | Yes | Project-level P&L; enables project profitability analysis by LLM |
| Contract ID | SaaS (ASC 606), Licensing | Yes | Revenue recognition context; LLM can trace revenue to contract terms |
| Channel | E-commerce, Retail, B2B | Yes | Direct vs. partner vs. marketplace revenue breakdown |
| Fund / Grant | Nonprofit, Government, Research | Yes | Restricted fund tracking; compliance reporting |
| Customer Segment | Any B2B | Sometimes | Customer cohort analysis; LTV/CAC by segment |

The critical takeaway: every NetSuite GL line can carry 7+ dimensions (account, subsidiary, department, class, location, period, plus custom segments). This is rich, multidimensional financial data that becomes extraordinarily powerful when properly denormalized and contextualized for LLM consumption.

## 1.2 Chart of Accounts and Projects: The GL’s Key Companions
While the GL records the transactions, two reference objects give those transactions their meaning:

### Chart of Accounts
The Chart of Accounts (CoA) is the master list of GL accounts that defines the structure of the financial statements. In NetSuite, each account carries: an account number (identifier, not meaningful in itself), an account name, an account type (one of NetSuite’s fixed types: Bank, Accounts Receivable, Other Current Asset, Fixed Asset, Other Asset, Accounts Payable, Credit Card, Other Current Liability, Long-Term Liability, Equity, Income, COGS, Expense, Other Income, Other Expense, Non-Posting), a parent account (for hierarchical rollup), and an optional description. The CoA is the single most important reference for LLM comprehension of financial data. When PipeLedger maps a company’s CoA to a standard taxonomy, it gives the LLM the ability to understand what every dollar in the GL represents.

### Project Records
For services companies, SaaS businesses tracking R&D, and any organization managing discrete initiatives, Project records in NetSuite provide a critical additional dimension. Projects can carry their own budgets, track time and expenses, and roll up to GL postings via the Project custom segment. PipeLedger extracts Project metadata (name, status, budget, start/end dates, project manager, customer association) and joins it to GL lines tagged with the project dimension. This enables project profitability analysis — one of the highest-value use cases for AI in professional services and R&D-heavy companies.

## 1.3 Complementary External Data: FP&A Systems
The one external data source in PipeLedger’s initial scope is the FP&A / budgeting system (Pigment, Adaptive Planning, Anaplan, or even a structured Excel/Google Sheets model). The FP&A system closely mirrors the GL: it contains account-level budget and forecast data organized by the same dimensions (account, department, class, period). This structural similarity makes it the natural first integration outside the ERP.


| FP&A Data Element | How It Maps to GL | Why It Matters for AI |
| --- | --- | --- |
| Budget amounts by account/department/period | Direct dimensional match to GL actuals | Budget vs. actual is the #1 most requested CFO analysis; LLMs can generate variance commentary automatically |
| Forecast / re-forecast amounts | Same structure as budget but updated quarterly or monthly | Enables AI to compare actuals against the latest forecast, not just the original annual budget |
| KPI targets (headcount, ARR, churn) | Mapped to GL accounts that drive these metrics | Provides the “why” behind financial targets; LLMs can explain whether the company is on track |
| Scenario models (best/base/worst) | Multiple budget versions against the same GL structure | AI can run sensitivity analysis: “If we’re tracking to the worst-case scenario, what changes?” |

Future scope (Year 2+): Notion/Confluence for qualitative context (project documentation, strategic rationale) and Jira/Linear for engineering effort allocation. These are deliberately excluded from v1 to maintain focus on the core GL-to-AI pipeline.

# Section 2: Data Transformations for the General Ledger
Raw GL data is structured for accountants, not for machines. The transformation layer is where PipeLedger adds its core value — converting ERP-native data structures into formats that LLMs can reason about reliably.

## 2.1 Universal Transformations


| Transformation | What It Does | Why LLMs Need It | dbt Model |
| --- | --- | --- | --- |
| Balance Decomposition | Strips opening/closing balances from GL exports; isolates period-over-period movements | LLMs cannot distinguish between a cumulative balance and a period movement; they need clean deltas to calculate changes | int_balance_decomposition |
| Account Taxonomy Mapping | Maps company-specific account numbers to a standard hierarchy (US GAAP / IFRS) | Without standard taxonomy, LLMs cannot compare accounts across companies or reason about account purpose | int_account_normalization |
| Dimension Denormalization | Flattens hierarchical dimensions into labeled attributes on each record | LLMs process flat records better than nested hierarchies; each row should be self-describing | int_dimension_flattening |
| Currency Standardization | Converts all amounts to reporting currency using period-end/average rates; tags original currency | LLMs will accidentally mix currencies if not standardized; explicit tagging prevents calculation errors | int_currency_conversion |
| Period Normalization | Aligns fiscal periods to a standard calendar (or clearly labels custom fiscal calendars) | LLMs assume calendar months; a company on a 4-4-5 calendar will confuse any model without explicit labeling | int_period_alignment |
| Context Enrichment | Adds natural language descriptions to account codes, dimension values, and transaction patterns | Transforms code-heavy records into self-describing records that LLMs can reason about without lookups | int_context_enrichment |

### Deep Dive: Account Taxonomy Mapping — How It Works Operationally
Account taxonomy mapping is the process of translating a company’s proprietary chart of accounts into a standardized hierarchy that any LLM (or any analyst) can immediately understand. This is not a one-time automated process — it requires human review and is unique to every organization.
Step 1: Ingest the CoA. PipeLedger extracts the full chart of accounts from NetSuite via SuiteQL, including account number, name, type, parent, and description. This becomes the source list.
Step 2: Auto-suggest mappings. PipeLedger’s taxonomy engine compares each account’s name, type, and description against the standard US GAAP / IFRS hierarchy and suggests the most likely mapping. For example, an account named “SaaS Subscription Revenue” with type “Income” is auto-mapped to “Revenue > Recurring Revenue > Subscription Revenue”. An account named “Cloud Hosting” with type “COGS” is auto-mapped to “Cost of Revenue > Infrastructure > Cloud Computing”.
Step 3: Human review in the UI. The PipeLedger Schemas page presents a two-column mapping interface: the customer’s accounts on the left, the standard taxonomy on the right. Each auto-suggested mapping is shown with a confidence score. The customer’s finance team reviews, corrects, and approves each mapping. Unmapped accounts are highlighted for manual assignment. This is the configuration step that makes PipeLedger company-specific.
Step 4: Version and lock. Once approved, the mapping is version-controlled (v1, v2, etc.). Changes require a new version with a review trail. The dbt model int_account_normalization reads the approved mapping table from Supabase and applies it to every GL record during transformation.
UI requirement: The mapping interface must support: search and filter on both sides, drag-and-drop mapping, bulk mapping for similar accounts, a “show unmapped only” filter, confidence scores on auto-suggestions, side-by-side comparison of the company’s hierarchy vs. the standard hierarchy, and an approval workflow where one person maps and another approves.

### Deep Dive: Dimension Denormalization — How It Works Operationally
Dimension denormalization converts the hierarchical, code-based dimension structure of the GL into flat, labeled attributes on every record. This is what makes the data self-describing for LLMs.
Step 1: Extract dimension hierarchies. PipeLedger extracts the full parent-child hierarchy for each dimension type (Department, Class, Location, and each custom segment) from NetSuite. For example, Department “210” has parent “200” (Engineering) which has parent “0” (root).
Step 2: Build the label path. For each dimension value, PipeLedger constructs the full hierarchy path as a human-readable string. Department 210 becomes “Engineering > Backend Infrastructure”. Class 12 becomes “SaaS > Enterprise > Annual Contracts”. This path is the label that appears on every denormalized GL record.
Step 3: Customer review of labels. The PipeLedger UI shows the auto-generated label paths alongside the raw dimension codes. Customers can override labels where the auto-generated path is misleading or where internal naming differs from external-facing names. For example, “Dept 300” might internally be called “RevOps” but the company prefers the LLM to see “Revenue Operations & Strategy”.
Step 4: Apply to every GL record. The dbt model int_dimension_flattening joins the dimension label table to every GL line, replacing code references with full label paths. The output record changes from “account_id: 4100, dept_id: 210, class_id: 12” to “account: SaaS Subscription Revenue, department: Engineering > Backend Infrastructure, class: SaaS > Enterprise > Annual Contracts”. Every record is now self-describing.
UI requirement: The dimension management interface must support: visual hierarchy tree view with expand/collapse, inline editing of labels, a preview mode showing how a sample GL record looks before and after denormalization, and the ability to add custom labels or descriptions at any level of the hierarchy.

## 3.2 Where Domain Knowledge Creates Defensible Value
The transformation rules themselves are the IP. Consider what it takes to build the balance decomposition model for NetSuite: you need to know that NetSuite’s GL Impact report includes cumulative balances, that the way it handles opening balances differs from SAP’s carry-forward logic, that the period boundary behavior changes depending on whether the company uses standard or 4-4-5 fiscal calendars, and that custom segments with GL impact behave differently from non-GL-impact segments in the balance computation. This knowledge comes from years of working inside these systems — not from reading API documentation.
PipeLedger applies domain knowledge in three tiers:
Tier 1: Structural Knowledge (hardcoded). How each ERP stores and exports GL data. The mapping between NetSuite’s internal transaction types and standard accounting entries. The rules for decomposing cumulative balances. This is baked into the connector and staging layers. An AI tool cannot replicate this without extensive ERP experience.
Tier 2: Accounting Standards Knowledge (configurable). The standard US GAAP / IFRS account taxonomy. Industry-specific templates (SaaS metrics, manufacturing cost accounting, project-based services). These are delivered as dbt model templates that the customer reviews and approves through the mapping UI.
Tier 3: Company-Specific Knowledge (customer-configured). Dimension label overrides, security policies, budget dimension mappings, and context enrichment rules unique to this company. PipeLedger never assumes or infers company-specific knowledge — the customer configures everything through the UI, and the dual-review checkpoint workflow ensures a human validates every output before delivery.
Legal framing: PipeLedger is a data transformation and enrichment tool, not an accounting service. Terms of service explicitly disclaim liability for the accuracy of source data. The customer is responsible for reviewing transformed output before delivery to any downstream system.

# Section 5: How LLMs Want Financial Data Presented
This section defines the target output format that all PipeLedger transformations aim to produce. Understanding how LLMs process structured data is essential to designing transformations that maximize AI reasoning quality.

## 5.1 Core Principles of LLM-Optimized Financial Data
Principle 1: Flat, Denormalized Records. LLMs process sequences of tokens. Hierarchical data structures (parent-child departments, nested account hierarchies) force the model to maintain state across multiple references. Flat records where every dimension is explicitly stated on every row dramatically improve comprehension. Instead of “Department: 200” (requiring a lookup), the record should read “Department: Engineering > Backend Infrastructure”.
Principle 2: Natural Language Context on Every Record. The single most impactful transformation PipeLedger performs is context enrichment. A GL line that reads “Account: 4100, Amount: $124,500 CR, Department: 100, Class: 10” is nearly useless to an LLM. The same record enriched to “Account: SaaS Subscription Revenue (4100), recognized $124,500 in January 2026, Sales department, Enterprise business line, from recurring contracts” enables the LLM to reason about the business meaning.
Principle 3: Movements, Not Balances. LLMs are not calculators. When given a balance of $500K followed by a balance of $620K, many LLMs will not reliably compute the $120K change. PipeLedger pre-computes all movements and presents them explicitly: “Marketing expense increased $120K (+24%) from December to January.”
Principle 4: Explicit Temporal Context. Every record must carry unambiguous time context. Not just “January” but “January 2026 (Month 7 of FY2026, Q3)”. This enables the LLM to understand seasonality, year-over-year comparisons, and fiscal calendar alignment without guessing.
Principle 5: Relationship Links Between Records. Financial data in the GL is not a flat table — it is a connected graph of relationships. PipeLedger must preserve and make explicit these relationships so LLMs can traverse them when answering complex questions. In the context of the NetSuite General Ledger with PipeLedger’s enrichment, the relationship graph looks like this:


| Relationship | How It Exists in NetSuite GL | How PipeLedger Exposes It | Example LLM Query It Enables |
| --- | --- | --- | --- |
| GL Transaction → Account | Each GL line references an account_id in the Chart of Accounts | Denormalized: account name, type, standard taxonomy mapping, and NL description embedded on every GL record | "What percentage of revenue comes from recurring sources?" |
| GL Transaction → Department | Each GL line carries a department_id (cost center) | Denormalized: full department hierarchy path as label (e.g., “Engineering > Backend”) on every record | "Which department is growing expenses fastest relative to budget?" |
| GL Transaction → Class | Each GL line carries a class_id (profit center) | Denormalized: full class hierarchy path (e.g., “SaaS > Enterprise > Annual”) on every record | "What is the gross margin by business line this quarter?" |
| GL Transaction → Project | GL lines tagged with project custom segment link to project records | Project metadata joined: project name, status, budget, manager, customer, start/end dates | "Which active projects are exceeding their cost budget?" |
| GL Transaction → Period | Each GL line is assigned to a fiscal period | Period metadata expanded: month name, quarter, fiscal year, days in period, period-end date | "How do Q3 expenses compare year-over-year?" |
| GL Transaction → Subsidiary | Each GL line belongs to a legal entity (OneWorld) | Subsidiary metadata: entity name, base currency, country, tax jurisdiction | "What is the revenue split across our US and EU entities?" |
| GL Actuals → Budget | No native link in NetSuite; lives in FP&A system | PipeLedger joins via aligned dimensions: same account + department + period = matched record | "Where are we most over/under budget this month?" |
| Account → Standard Taxonomy | No native standard in NetSuite; company-specific CoA only | PipeLedger’s taxonomy mapping creates the bridge to US GAAP / IFRS standard hierarchy | "Categorize our expenses by standard GAAP classification" |

Data lineage summary for NetSuite GL with enrichment: Every output record PipeLedger delivers to the LLM carries a complete lineage chain: the original NetSuite transaction ID and line number (provenance), the account with both the company’s code and the standard taxonomy mapping (classification), all dimensions denormalized with full hierarchy labels (context), the fiscal period with both NetSuite’s period and the normalized calendar period (temporal), the currency with both original and converted amounts plus the FX rate used (monetary), the budget amount for the same dimension intersection if available (planning), and a natural language description summarizing the record’s business meaning (comprehension). This means the LLM can not only answer the question but trace the answer back to the source transaction in NetSuite — and the human reviewer can verify it.
Principle 6: Appropriate Granularity with Summary Context. LLMs have context window limits. Sending 2 million individual GL lines is not practical. PipeLedger provides data at multiple granularities: summary-level for broad questions (“What were total expenses?”), account-level for variance analysis (“Which expense categories grew?”), and transaction-level for investigation (“What drove the marketing spike?”). The MCP server intelligently selects the appropriate granularity based on the query.

## 7.1 Monthly Close: Security, Approval & Controlled Delivery
The problem: Today, when a finance team exports GL data to share with an LLM, board members, or external advisors, there is no systematic control over what data is included. Sensitive GL lines (executive compensation, M&A project costs, board-related entries) are mixed in with operational data. The team either shares everything (risky) or manually redacts line by line (slow and error-prone).
With PipeLedger: Before any transformed GL data is delivered to an LLM or exported, it passes through PipeLedger’s security and approval pipeline:
Row-Level Security: Customer-defined rules automatically tag sensitive GL records based on account ranges (e.g., accounts 6100–6199 = compensation), dimension values (e.g., Project = “M&A Due Diligence”), or custom tags. When the data is delivered via MCP or API, redacted records are replaced with masked placeholders (“██████ — Restricted”) and the LLM receives a count of redacted records rather than attempting to access them.
Dual-Checkpoint Approval: The pipeline pauses at two points: once after extraction (input review — “did we pull the right data from NetSuite?”) and once after transformation (output review — “does the enriched data look correct before we ship it?”). An Approver-level user must sign off at each checkpoint. This creates an auditable chain of custody from ERP to LLM.
Controlled delivery: Once approved, the data is delivered only to the configured endpoints (MCP server, API, file export). The finance team maintains full control over who can access what, at what granularity, through which channel.

# Section 8: PipeLedger’s Core Pipeline — Ingest, Transform, Deliver
PipeLedger’s product is a three-stage pipeline. Every feature, every transformation, and every use case described in this document maps to one of these three stages. This is the product — nothing more, nothing less.

## 8.1 Stage 1: Data Ingestion


| Component | What It Does | Technology | Customer Interaction |
| --- | --- | --- | --- |
| NetSuite Connector | Extracts GL transactions, Chart of Accounts, dimensions (departments, classes, locations, custom segments), project records, and subsidiary metadata via SuiteQL API | TypeScript, SuiteQL, paginated extraction with change-tracking | Customer provides NetSuite credentials; PipeLedger configures which data objects to extract |
| FP&A Import | Ingests budget/forecast data from FP&A systems (Pigment, Adaptive, Anaplan) or structured CSV/Excel files | CSV parser, API integrations (future), Supabase storage | Customer uploads budget file or connects FP&A API; PipeLedger validates format and dimensions |
| Extraction Scheduling | Runs ingestion on configurable schedules (daily, weekly, manual) via Dagster pipeline orchestration | Dagster sensors and schedules, Cloud Run worker | Customer selects schedule; Dagster triggers extraction at configured times |
| Input Review Checkpoint | Pauses pipeline after extraction; presents extracted record counts and sample data for human review before proceeding | Supabase (review_checkpoints table), Next.js review UI | Approver reviews extracted data, confirms it looks correct, approves to proceed |

## 8.2 Stage 2: Transformation


| Component | What It Does | Technology | Customer Interaction |
| --- | --- | --- | --- |
| Balance Decomposition | Strips cumulative balances, computes period movements by account/dimension | dbt model: int_balance_decomposition running in BigQuery | None — automatic; logic is ERP-specific and hardcoded |
| Account Taxonomy Mapping | Applies customer-approved mapping from their CoA to standard US GAAP / IFRS hierarchy | dbt model: int_account_normalization reading mapping table from Supabase | Customer maps accounts in the Schemas UI; mapping is versioned and approved |
| Dimension Denormalization | Resolves hierarchical dimension codes into flat labeled paths on every GL record | dbt model: int_dimension_flattening reading dimension labels from Supabase | Customer reviews and overrides auto-generated labels in the dimension management UI |
| Currency Standardization | Converts all amounts to reporting currency using configured FX rates | dbt model: int_currency_conversion using rate tables in BigQuery | Customer configures reporting currency and rate source (market rates or manual) |
| Period Alignment | Normalizes fiscal periods to calendar months (or labels custom fiscal periods explicitly) | dbt model: int_period_alignment using fiscal calendar config from Supabase | Customer confirms fiscal calendar setup; auto-detected from NetSuite |
| Context Enrichment | Adds natural language descriptions to every record based on account type, dimensions, and amount patterns | dbt model: int_context_enrichment using taxonomy + dimension labels + amount metadata | None — automatic; uses approved mappings and labels from previous steps |
| Row-Level Security Tagging | Tags each GL record with security classification based on customer-defined rules | dbt model: sec_rls_tags reading security policies from Supabase | Customer defines security rules in Settings UI: account ranges, dimensions, project codes |
| Budget vs. Actual Join | Joins FP&A budget data to GL actuals by aligned dimensions; computes variances and context | dbt model: int_budget_vs_actual joining stg_budget to mart_gl_movements | Customer maps FP&A dimensions to GL dimensions in the mapping UI |

## 8.3 Stage 3: Delivery


| Component | What It Does | Technology | Customer Interaction |
| --- | --- | --- | --- |
| Output Review Checkpoint | Pauses pipeline after transformation; presents quality checks, data preview (with RLS applied), and reconciliation summary for human approval | Supabase (review_checkpoints), Next.js review UI with “Viewing as” role selector | Approver reviews transformed data at their access level, confirms quality, approves delivery |
| MCP Server | Serves transformed financial data to Claude and other LLMs as discoverable resources and tools via Model Context Protocol | TypeScript MCP SDK, Next.js API routes, BigQuery read access | Customer configures which data is exposed; access controlled by API key and RLS |
| REST API | Provides programmatic access to transformed data as JSON/JSONL with full dimension context and NL descriptions | Next.js API routes, BigQuery read access, Redis cache | Customer generates API keys; access scoped by organization and role |
| File Export (Parquet/CSV) | Generates columnar (Parquet) or flat (CSV) files of transformed data for batch consumption, ML training, or data warehousing | TypeScript export worker, Google Cloud Storage staging | Customer configures export format, schedule, and destination (GCS bucket or download) |
| Audit Trail | Records every pipeline run, every approval, every data access, and every delivery with user, timestamp, and changes | Supabase (audit_logs), immutable append-only table | Visible in Activity page; exportable for compliance purposes |

It transforms the data using deep accounting domain knowledge encoded as dbt models. It delivers the result — clean, contextualized, security-controlled — to LLMs, APIs, and files. The customer reviews and approves at every stage. The audit trail records everything. No more, no less. The Stripe of financial data.
